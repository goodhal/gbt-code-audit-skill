# 代码安全审计技能

## 什么时候使用

当你需要对代码进行安全审计，检测潜在的安全漏洞，确保代码符合中国国家标准时使用此技能。适用于代码开发、代码审查、安全评估等场景。

## 安装设置

- Python 3.7+

## 功能特性

- **多语言支持**: Java、C/C++、C#、Python（LLM 自主判断）
- **国标覆盖**: GB/T 34943-2017 (C/C++)、GB/T 34944-2017 (Java)、GB/T 34946-2017 (C#)、GB/T 39412-2020 (通用)
- **快速扫描**: 正则表达式模式匹配检测高危漏洞
- **智能去重**: 数据库中按规则去重，合并相同问题
- **标准报告**: 按国标章节分类生成合规审计报告

```
┌─────────────────────────────────────────────────────────────────────┐
│                        Agent 代码安全审计流程                        │
├─────────────────────────────────────────────────────────────────────┤
│  0️⃣ 学习流程  ◀ 【强制】阅读 SKILL.md                               │
│  1️⃣ 语言判定  ◀ 【LLM 自主】通过文件扩展名/内容判断语言              │
│  2️⃣ 学习标准  ◀ 【必须输出】确认已学习的国标及规则数                 │
│  3️⃣ 初始化数据库  ◀ 【必须调用 init_report】创建数据库              │
│  4️⃣ 快速扫描  ◀ 【必须调用 quick_scan】获取基线结果                  │
│  5️⃣ 基线入库  ◀ 【必须】创建md文件                                  │
│  6️⃣ LLM 深度审计  ◀ 【必须】创建md文件 + validate_finding验证       │
│  7️⃣ 生成报告  ◀ 【必须调用 finalize_report】去重+验证+生成报告      │

```

## 工具列表

| 工具名称 | 描述 | 命令行用法 |
|---------|------|----------|
| `quick_scan` | 快速扫描：正则表达式模式匹配检测常见漏洞 | `python skill.py quick_scan <target>` |
| `extract_code` | 提取代码：获取指定文件和行号的真实代码片段 | `python skill.py extract_code <file_path> <line_number> [--context=3]` |
| `validate_finding` | 验证发现：验证md文件的代码片段是否真实存在（防幻觉） | `python skill.py validate_finding <md_file_path>` |
| `finalize_report` | 收尾报告：遍历 md 文件去重 + 验证幻觉 + 生成报告 + 清空 findings 目录 | `python skill.py finalize_report [...]` |

> 💡 **防幻觉机制**：
> - **validate_finding（推荐）**：LLM创建md后立即验证，失败则修改内容再验证
> - **finalize_report（兜底）**：报告生成时再次验证，过滤遗漏的幻觉问题

## Markdown文件格式

每个审计发现以Markdown文件形式存储，文件命名格式：`findings/{source}/{编号}.md`

**目录结构**：

```
findings/
├── baseline/      # 快速扫描基线结果
│   ├── 001.md
│   ├── 002.md
│   └── ...
└── llm_audit/     # LLM深度审计结果
    ├── 001.md
    ├── 002.md
    └── ...
```

**文件内容格式**：

```markdown
# 审计发现 #编号

编号: #001
严重等级: 严重
漏洞类型: 命令注入
文件路径: test-samples/java/VulnerableJava.java
行号: 49
CWE: CWE-78
国标映射: GB/T34944-6.2.3.3 命令注入
来源: quick_scan
语言: java
问题代码: Runtime.getRuntime().exec("cat " + filename);
问题描述: 使用未经验证的输入数据构建命令，攻击者可执行任意命令
修复方案: 使用ProcessBuilder并设置参数白名单验证
验证方法: 传入恶意参数如"test.txt; rm -rf /"验证命令执行
```

**字段说明**：

| 字段 | 必填 | 说明 |
|------|------|------|
| 编号 | ✅ | 格式：#001 |
| 严重等级 | ✅ | 严重/高危/中危/低危 |
| 漏洞类型 | ✅ | 中文名称 |
| 文件路径 | ✅ | 相对路径 |
| 行号 | ✅ | 问题代码所在行号 |

**⚠️ 行号填写要求**：
- **必须使用 Read/Grep 工具验证行号**
- **行号必须对应实际代码行，不能是注释行**
- **禁止凭记忆或推断填写行号**
- **每个漏洞必须有唯一行号**
| CWE | ✅ | 如 CWE-78 |
| 国标映射 | ✅ | 格式：GB/T标准号-章节号 规则名称 |
| 来源 | ✅ | quick_scan 或 llm_audit |
| 语言 | ✅ | java/cpp/csharp/python |
| 问题代码 | ✅ | 问题代码片段 |
| 问题描述 | ✅ | 2-3句话描述 |
| 修复方案 | ✅ | 可执行的修复代码或方案 |
| 验证方法 | ✅ | 如何验证漏洞 |

## 步骤详解

### 步骤 3：创建 findings 目录

**由 LLM 直接创建**（无需调用工具）：

```powershell
New-Item -ItemType Directory -Force -Path findings\baseline
New-Item -ItemType Directory -Force -Path findings\llm_audit
```

### 步骤 4：快速扫描

```powershell
python skill.py quick_scan test-samples
```

### 步骤 5：基线入库

**任务**：严格按照快速扫描输出结果创建 Markdown 文件

**⚠️ 强制要求**：
- **必须严格遵循 quick_scan 的输出结果**
- **每个快速扫描发现对应一个 md 文件**
- **禁止额外添加或遗漏任何发现**
- **文件数量必须等于 quick_scan 返回的 total_findings 数**

**🔴 基线入库验证流程（必须严格执行）**：

```
┌─────────────────────────────────────────────────────────────────────┐
│                    基线入库验证流程                                  │
├─────────────────────────────────────────────────────────────────────┤
│  1️⃣ 运行快速扫描  ◀ 【必须】执行 quick_scan 获取 JSON 输出          │
│  2️⃣ 解析JSON输出  ◀ 【必须】解析 findings 数组和 total_findings    │
│  3️⃣ 按顺序创建文件  ◀ 【必须】按 findings 数组顺序创建 md 文件      │
│  4️⃣ 验证文件数量  ◀ 【必须】确认文件数等于 total_findings          │
│  5️⃣ 验证文件内容  ◀ 【必须】确认每个文件对应一个发现               │
└─────────────────────────────────────────────────────────────────────┘
```

**执行方式**：
1. **【必须】运行快速扫描**：
   ```bash
   python skill.py quick_scan test-samples
   ```
2. **【必须】解析 JSON 输出**：
   - 提取 `findings` 数组
   - 提取 `total_findings` 数值
   - 记录每个发现的 `file`、`line`、`type`、`cwe`、`severity` 等字段
3. **【必须】按顺序创建文件**：
   - 文件编号从 001 开始顺序递增（001.md, 002.md, ...）
   - 文件编号必须与 findings 数组索引对应
   - **禁止跳过、重复或额外添加发现**
4. **【必须】验证文件数量**：
   - 创建完成后，统计 `findings/baseline/` 目录下的 md 文件数量
   - 文件数量必须等于 `total_findings`
   - 如果不一致，必须检查并修正

**⚠️ 常见错误及预防**：

| 错误类型 | 错误示例 | 正确做法 |
|---------|---------|---------|
| 文件数量不符 | quick_scan返回29，创建了34个文件 | 必须严格按findings数组创建 |
| 编号不连续 | 跳过了005.md，从004直接到006 | 必须连续编号，不能跳过 |
| 内容不匹配 | 003.md内容与findings[2]不一致 | 必须按数组顺序对应创建 |
| 额外添加发现 | 添加了quick_scan未发现的漏洞 | 禁止额外添加，仅限quick_scan结果 |

### 步骤 6：LLM 深度审计

**任务**：独立完整审计所有源文件，为每个发现创建 Markdown 文件

**🔴 核心原则：LLM审计必须完全独立**
- **❌ 禁止参考快速扫描结果**：LLM审计是独立完整的审计过程，不得查看或参考快速扫描的输出
- **❌ 禁止复用基线发现**：必须独立发现漏洞，不得复制baseline目录中的内容
- **✅ 必须独立分析**：逐行阅读源代码，独立判断每个安全问题
- **✅ 结果可能不同**：LLM审计结果数量通常与快速扫描不同，这是正常的

**⚠️ 强制要求**：
- **必须逐行核对源文件确认漏洞位置**
- **行号必须精确对应问题代码所在行**
- **禁止根据注释推断行号，必须定位实际代码**
- **每个漏洞的行号必须唯一且准确**
- **必须使用 Grep 工具精确查找行号，禁止凭记忆填写**

**🔴 行号验证流程（必须严格执行）**：

```
┌─────────────────────────────────────────────────────────────────────┐
│                    LLM 审计行号验证流程                              │
├─────────────────────────────────────────────────────────────────────┤
│  1️⃣ 读取源文件  ◀ 【必须】使用 Read 工具读取完整源文件              │
│  2️⃣ 定位问题代码  ◀ 【必须】在源文件中精确搜索问题代码片段          │
│  3️⃣ 确认行号  ◀ 【必须】验证行号对应的是代码行而非注释行            │
│  4️⃣ 二次验证  ◀ 【推荐】使用 extract_code 工具验证代码片段         │
│  5️⃣ 记录发现  ◀ 【必须】行号验证通过后才创建 md 文件               │
└─────────────────────────────────────────────────────────────────────┘
```

**🆕 使用 extract_code 工具验证（推荐）**：

在创建 md 文件前，使用 `extract_code` 工具验证代码片段的真实性：

```powershell
# 验证指定文件和行号的代码
python skill.py extract_code test-samples/python/vulnerable_python.py 62

# 输出示例：
# {
#   "success": true,
#   "file": "test-samples/python/vulnerable_python.py",
#   "line": 62,
#   "code_snippet": "subprocess.call(\"ls -la \" + filename, shell=True)  # shell=True危险",
#   "context": "...上下文代码...",
#   "total_lines": 638
# }
```

**验证要点**：
- `code_snippet` 必须与你准备写入 md 文件的问题代码匹配
- `line` 必须是你报告的行号
- 如果 `success: false`，说明行号超出范围或文件不存在

**执行方式**：
1. **【必须】读取所有源文件**：使用 Read 工具逐个读取源文件
2. **【必须】逐行分析代码**：对照国标规则逐条检查
3. **【必须】精确定位问题代码行号**：
   - 使用 Grep 工具搜索问题代码片段
   - 确认搜索结果的行号
   - 读取该行号附近内容验证
4. **【必须】排除注释行**：
   - 行号对应的行不能是注释行（以 # 或 // 开头）
   - 行号对应的行不能是空行
   - 行号必须对应实际执行的代码
5. **【必须】验证每个发现**：
   
   **防幻觉工作流程**：
   ```
   ┌─────────────────────────────────────────────────────────────────────┐
   │                    LLM 审计防幻觉流程                               │
   ├─────────────────────────────────────────────────────────────────────┤
   │  1️⃣ 创建md文件  ◀ LLM写入审计发现到 findings/llm_audit/XXX.md     │
   │  2️⃣ 调用验证    ◀ python skill.py validate_finding XXX.md         │
   │  3️⃣ 验证结果    ◀ 成功 → 编号+1，开始下一个发现                    │
   │               ◀ 失败 → 尝试修正行号 → 成功则通过，失败则用下一个问题覆盖 → 再次验证 │
   └─────────────────────────────────────────────────────────────────────┘
   ```
   
   **⚠️ 必须逐个验证**：
   - **❌ 禁止批量验证**：创建多个md文件后一次性验证是错误的
   - **✅ 必须一个一个验证**：创建一个md → 验证 → 通过后再创建下一个
   - **✅ 验证通过才能继续**：只有验证成功后才能开始下一个发现
   
   **⚠️ 验证失败处理**：
- **第一步**：尝试修正行号，使用Grep重新搜索正确的代码位置
- **第二步**：如果找到正确行号，更新md文件后再次验证
- **第三步**：如果无法找到正确行号，用下一个问题覆盖当前md文件内容后再次验证
- **禁止**：反复修改同一个问题超过2次，超过则跳过该问题
   
   **验证命令**：
   ```powershell
   python skill.py validate_finding findings/llm_audit/001.md
   ```
   
   **验证成功输出**：
   ```json
   {
     "success": true,
     "md_file": "findings/llm_audit/001.md",
     "validation": "matched",
     "message": "验证通过，可以开始下一个发现"
   }
   ```
   
   **验证失败输出**：
   ```json
   {
     "success": false,
     "error": "代码片段验证失败，可能存在幻觉",
     "actual_code": "实际代码内容...",
     "hint": "尝试修正行号，若无法修正则用下一个问题覆盖当前md"
   }
   ```

6. **所有发现入库**（不去重，包括基线已包含的问题）

**⚠️ 常见幻觉问题及预防**：

| 幻觉类型 | 错误示例 | 正确做法 |
|---------|---------|---------|
| 注释行号 | 报告行号27，实际是注释行 | 必须定位到实际代码行 |
| 行号偏移 | 报告行号150，实际代码在133 | 使用Grep精确搜索 |
| 代码不存在 | 报告的代码片段在源文件中不存在 | 必须在源文件中验证存在 |
| 重复行号 | 多个漏洞使用相同行号 | 每个漏洞必须有唯一行号 |

**⚠️ 重要提示**：
- LLM审计是独立完整的审计，不要跳过任何问题
- 每个问题独立存储为一个Markdown文件
- **行号准确性是报告质量的关键指标**
- **禁止凭记忆或推断填写行号，必须实际验证**

### 步骤 7：生成报告

**自动生成带时间戳的报告文件名**：

```powershell
python skill.py finalize_report --project=test-samples --languages=java,cpp,csharp,python --standards=GB/T34944,GB/T34943,GB/T34946,GB/T39412 --date=2026-04-16
```

**输出文件**：`audit_report_YYYYMMDD_HHMMSS.md`（自动添加时间戳）

**自定义输出文件名**：

```powershell
python skill.py finalize_report --output=my_custom_report.md --project=test-samples
```

**自动执行**：
- 内存去重（按 `{文件路径}:{行号}:{漏洞类型}` 去重）
- 优先保留 LLM 审计结果
- 动态生成报告
- 清空 findings 目录（验证成功后）
