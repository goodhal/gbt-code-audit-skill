# 代码安全审计技能

> **测试环境**：已在 GLM-5 模型下测试通过

## 📋 概述

本技能提供完整的代码安全审计工作流，基于中国国家标准检测源代码中的安全漏洞。

**核心流程**：
```
快速扫描 (正则) → 基线入库 (md 文件) → LLM 深度审计 → 内存去重 → 生成 Markdown 报告
```

**输出物**：
- `findings/baseline/*.md` — 快速扫描发现的漏洞
- `findings/llm_audit/*.md` — LLM 独立审计发现的漏洞
- `audit_report_YYYYMMDD_HHMMSS.md` — 最终审计报告（含汇总表格和详细发现）

**关键机制**：
- **防幻觉**：每个 LLM 发现必须通过 `validate_finding` 验证代码片段真实存在
- **独立性**：LLM 审计时禁止查看快速扫描结果，必须逐行独立分析
- **去重**：按 `{文件路径}:{行号}:{漏洞类型}` 去重，优先保留 LLM 审计结果

---

## 什么时候使用

当你需要对代码进行安全审计，检测潜在的安全漏洞，确保代码符合中国国家标准时使用此技能。适用于代码开发、代码审查、安全评估等场景。

## 安装设置

- Python 3.7+

## 功能特性

- **多语言支持**: Java、C/C++、C#、Python（LLM 自主判断）
- **国标覆盖**: GB/T 34943-2017 (C/C++)、GB/T 34944-2017 (Java)、GB/T 34946-2017 (C#)、GB/T 39412-2020 (通用)
- **快速扫描**: 正则表达式模式匹配检测高危漏洞
- **智能去重**: 内存中按规则去重（文件 + 行号 + 类型），合并相同问题
- **标准报告**: 按国标章节分类生成合规审计报告

### 1. LLM 审计独立性（最高优先级）

**❌ 绝对禁止**：
1. 查看快速扫描结果（findings/quick_scan/ 或 findings/baseline/）
2. 参考基线发现（baseline/目录内容）
3. 知道每个文件已有多少个发现
4. 使用"补充""另外""还有""共 X 个"等暗示参考的词汇
5. 在 LLM 审计文件中提及快速扫描

**✅ 必须执行**：
1. 从零开始逐行阅读源代码
2. 独立判断每个安全问题
3. 接受结果可能与快速扫描重复（会被 finalize_report 去重）
4. 结果数量完全不确定（可能 0 个，可能 20 个）

**⚠️ 违反后果**：审计结果无效，必须重新执行整个流程

### 2. 步骤执行强制性

**❌ 禁止跳过**：
1. 禁止跳过 SKILL.md 中任何编号步骤
2. 禁止自行模拟工具输出（必须实际调用 skill.py）
3. 禁止合并步骤（如基线入库 + LLM 审计一起做）

**✅ 必须确认**：
1. 每步完成后输出确认清单
2. 关键步骤前回答检查清单问题
3. 验证失败立即停止并修正

## Markdown文件格式

每个审计发现以Markdown文件形式存储，文件命名格式：`findings/{source}/{编号}.md`

**目录结构**：

```
findings/
├── baseline/      # 快速扫描基线结果
│   ├── 001.md
│   ├── 002.md
│   └── ...
└── llm_audit/     # LLM深度审计结果
    ├── 001.md
    ├── 002.md
    └── ...
```

**文件内容格式**：

```markdown
# 审计发现 #编号

编号: #001
严重等级: 严重
漏洞类型: 命令注入
文件路径: test-samples/java/VulnerableJava.java
行号: 49
CWE: CWE-78
国标映射: GB/T34944-6.2.3.3 命令注入
来源: quick_scan
语言: java
问题代码: Runtime.getRuntime().exec("cat " + filename);
问题描述: 使用未经验证的输入数据构建命令，攻击者可执行任意命令
修复方案：1) 改用 ProcessBuilder 参数数组：`new ProcessBuilder("cat", filename).start()`；2) 白名单验证：`filename.replaceAll("[^a-zA-Z0-9_.]", "")`；3) 过滤特殊字符 `|&;$`\`
验证方法: 传入恶意参数如"test.txt; rm -rf /"验证命令执行
```

**字段说明**：

| 字段 | 必填 | 说明 |
|------|------|------|
| 编号 | ✅ | 格式：#001 |
| 严重等级 | ✅ | 严重/高危/中危/低危 |
| 漏洞类型 | ✅ | 中文名称 |
| 文件路径 | ✅ | 相对路径 |
| 行号 | ✅ | 问题代码所在行号 |

**⚠️ 行号填写要求**：
- **必须使用 Read/Grep 工具验证行号**
- **行号必须对应实际代码行，不能是注释行**
- **禁止凭记忆或推断填写行号**
- **每个漏洞必须有唯一行号**
| CWE | ✅ | 如 CWE-78 |
| 国标映射 | ✅ | 格式：GB/T 标准号 - 章节号 规则名称；多标准用分号分隔 |
| 来源 | ✅ | quick_scan 或 llm_audit |
| 语言 | ✅ | java/cpp/csharp/python |
| 问题代码 | ✅ | 问题代码片段 |
| 问题描述 | ✅ | 2-3句话描述 |
| 修复方案 | ✅ | 可执行的修复代码或方案 |
| 验证方法 | ✅ | 如何验证漏洞 |

## 🔴 输出质量检查（防止敷衍 - 2026-04-18 更新）

### 修复方案编写要求

**❌ 禁止以下敷衍内容（出现则验证失败）**：
1. "根据国标要求修复"
2. "消除安全隐患"
3. "使用安全的方法"
4. "加强验证"
5. "进行过滤"
6. "根据相关标准处理"
7. "修复安全漏洞"
8. 字数 < 30 字
9. 不包含具体代码、命令、配置或 API 名称
10. 不包含可执行的技术方案

**✅ 必须包含以下内容**：
1. **具体代码示例**（如 `PreparedStatement ps = conn.prepareStatement("SELECT * FROM users WHERE id = ?");`）
2. **或具体命令**（如 `chmod 600 /etc/secret`）
3. **或具体 API/类名**（如 `Cipher.getInstance("AES/GCM/NoPadding")`、`BCrypt.hashpw()`）
4. **或具体配置参数**（如 `-fstack-protector`、`TLS 1.2+`）
5. **字数 ≥ 30 字**
6. **必须包含技术动作词**：如"使用"、"改用"、"替代"、"配置"、"启用"等

**修复方案示例库**：

| 漏洞类型 | ✅ 合格修复方案 | ❌ 不合格修复方案 |
|---------|----------------|-----------------|
| SQL 注入 | 使用 PreparedStatement：`ps = conn.prepareStatement("SELECT * FROM users WHERE id = ?"); ps.setString(1, userId);` 禁止字符串拼接 SQL。 | 根据国标要求修复，消除安全隐患。 |
| 命令注入 | 改用 ProcessBuilder 参数数组：`new ProcessBuilder("cat", filename).start()` 并对 filename 白名单验证仅允许字母数字。 | 使用安全的方法执行命令。 |
| 硬编码密码 | 密码移至环境变量：`System.getenv("DB_PASSWORD")` 或使用配置文件（权限 600），口令存储使用 bcrypt 哈希。 | 加强密码管理。 |
| 弱加密 | 使用 AES-256/GCM 替代 DES：`Cipher.getInstance("AES/GCM/NoPadding")` 密钥至少 256 位，使用 SecureRandom 生成。 | 使用更安全的加密算法。 |
| 弱哈希 | 使用 SHA-256 或 SM3 替代 MD5/SHA-1，口令哈希使用 bcrypt：`BCrypt.hashpw(password, BCrypt.gensalt(12))`。 | 进行过滤。 |
| 缓冲区溢出 | 使用 strncpy 替代 strcpy：`strncpy(buf, input, sizeof(buf)-1); buf[sizeof(buf)-1] = '\0';` 启用编译器保护 `-fstack-protector`。 | 加强边界检查。 |
| 反序列化 | 禁止 pickle.loads() 反序列化不受信任数据，改用 JSON，或实现白名单类限制可反序列化的类。 | 进行验证。 |

### 修复方案质量验证（双保险机制）

**第一道：LLM 自主检查**（创建 md 时执行）

LLM 创建 md 文件时必须自行检查修复方案质量：

| 检查项 | 要求 | 示例 |
|--------|------|------|
| 字数 | ≥ 30 字 | ❌ "根据国标修复" → ✅ 提供具体代码 |
| 禁用词 | 不含"根据国标""消除隐患""加强""进行过滤" | ❌ "使用安全的方法" → ✅ "改用 ProcessBuilder" |
| 技术关键词 | 包含 `()`、`=`、`:` 或"使用""改用""替代""配置" | ❌ "加强验证" → ✅ "使用 PreparedStatement" |

**第二道：validate_finding 函数验证**（程序化检查）

`validate_finding` 函数（skill.py 第 1113-1143 行）会自动检查修复方案质量，验证失败将拒绝该 md 文件：
- 字数 < 30 → 拒绝
- 包含禁用词 → 拒绝
- 不含技术关键词 → 拒绝

> 💡 **双保险**：LLM 自主检查 + 函数程序化验证，确保修复方案具有可执行性。

### 步骤 4：快速扫描

```powershell
python skill.py quick_scan test-samples
```

### 步骤 5：基线入库

**任务**：严格按照快速扫描输出结果创建 Markdown 文件

**⚠️ 强制要求**：
- **必须严格遵循 quick_scan 的输出结果**
- **每个快速扫描发现对应一个 md 文件**

1. **每个问题的代码片段必须真实存在**
2. **修复代码必须是可执行的**
3. **国标映射必须准确**（规则号 + 规则名称）
4. **CWE 编号必须与漏洞类型匹配**
5. **详细条目数必须与统计总数一致**
6. **必须执行基线入库步骤**：创建 md 文件
7. **必须执行 LLM 深度审计**：创建 md 文件
8. **去重由 finalize_report 自动执行**
9. **必须验证行号准确性**：使用 Read/Grep 工具实际验证
10. **行号必须对应实际代码行**：不能是注释行或空行
11. **每个漏洞必须有唯一行号**：禁止多个漏洞使用相同行号

## 🔧 常见问题与解决方案

### 问题 0：文档与代码不一致（执行前必查）

**症状**：
- SKILL.md 中提到 `init_report` 函数但 skill.py 中不存在
- 流程图要求"初始化数据库"但代码使用 Markdown 文件存储
- 命令示例使用 PowerShell 但环境是 Linux/bash

**原因**：
- 技能架构从 SQLite 数据库方案迁移到 Markdown 文件方案
- 文档更新滞后于代码变更

**解决方案**：
1. 执行前用 `grep -n "def " skill.py` 确认关键函数存在
2. 发现不一致时先修复 SKILL.md 再执行审计
3. 当前正确流程：创建目录 → 快速扫描 → 基线入库 → LLM 审计 → 生成报告

**已修复的不一致**：
- ~~初始化数据库~~ → 创建 findings 目录 (`mkdir -p findings/baseline findings/llm_audit`)
- ~~数据库去重~~ → 内存去重（`deduplicate_findings` 函数）
- ~~PowerShell 命令~~ → bash 命令

### 问题 1：md 文件无法解析，报告只显示 0 或 1 个发现

**症状**：
- `finalize_report` 输出 `total_findings: 0` 或 `total_findings: 1`
- 但 `findings/` 目录下有多个 md 文件

**原因**：
- md 文件使用了中文冒号 `：` (Unicode 0xff1a) 而不是英文冒号 `:` (ASCII 0x3a)
- `parse_finding_md` 和 `validate_finding` 函数需要正确解析键值对

**解决方案**：

**✅ 推荐格式**（英文键名 + 英文冒号，兼容性最好）：
```markdown
id: #001
severity: 严重
file: test-samples/java/VulnerableJava.java
line: 31
cwe: CWE-78
gbt_mapping: GB/T34944-6.2.3.3 命令注入
source: quick_scan
language: java
code_snippet: Runtime.getRuntime().exec(command);
description: 描述内容
fix: 修复内容
verification: 验证内容
```

**✅ 支持格式**（中文键名 + 英文冒号）：
```markdown
编号：#001
严重等级：严重
文件路径：test-samples/java/VulnerableJava.java
行号：31
CWE: CWE-78
国标映射：GB/T34944-6.2.3.3 命令注入
来源：quick_scan
语言：java
问题代码：Runtime.getRuntime().exec(command);
问题描述：描述内容
修复方案：修复内容
验证方法：验证内容
```

**⚠️ 2026-04-18 更新**：`validate_finding` 函数已修复为同时支持中文冒号和英文冒号。但仍推荐使用英文冒号以保证最大兼容性。

**⚠️ 2026-04-18 更新**：最终报告（`finalize_report` 生成）的每个问题明细必须包含"**严重性**"字段：
```markdown
### #1 🔴 命令注入
**来源**: 🔧 快速扫描
**严重性**: 严重    ← 必须包含此字段
**文件**: test-samples/java/VulnerableJava.java:31
```
严重性值必须为：严重 / 高危 / 中危 / 低危（与标题行图标🔴/🟠/🟡/🟢对应）

**批量修复命令**（如果验证失败）：
```bash
sed -i 's/：/:/g' findings/*/*.md
```

### 问题 2：LLM 审计发现被过滤（幻觉检测）

**症状**：
- `finalize_report` 输出警告：`检测到 X 个幻觉问题，已过滤`
- `hallucinations` 数组显示 `reason: mismatch`

**原因**：
- LLM 编造了不存在的行号或代码片段
- `validate_code_snippet` 函数验证失败

**解决方案**：
1. 创建 md 文件前，使用 `extract_code` 工具验证行号：
   ```bash
   python skill.py extract_code test-samples/java/VulnerableJava.java 31
   ```
2. 或使用 grep 精确搜索代码：
   ```bash
   grep -n "Runtime.getRuntime" test-samples/java/VulnerableJava.java
   ```
3. 确保行号对应实际代码行，不是注释行

### 问题 3：报告生成后详细发现为空

**症状**：
- `validation.success: false`
- `detailed_count: 0` 但 `total_count: 36`
- 报告文件只有统计表，没有详细发现

**原因**：
- 输出文件名包含空格或中文字符，被错误解析为参数
- 模板占位符替换失败

**解决方案**：
```bash
# ✅ 正确：使用英文文件名
python skill.py finalize_report --output=审计报告.md --project=测试项目

# ❌ 错误：空格导致参数解析错误
python skill.py finalize_report --output=审计报告_最终版.md --project=gbt-code-audit-skill 测试样例
```

### 问题 4：快速扫描结果为 0 或失败

**症状**：
- `quick_scan` 返回 `total_findings: 0`
- 或 `success: false, error: Java command not found`

**原因**：
- 目标目录不存在或没有源代码文件
- （仅 Java）缺少 JDK 环境

**解决方案**：
1. 确认目标路径正确：
   ```bash
   ls test-samples/java/ test-samples/python/ test-samples/cpp/ test-samples/csharp/
   ```
2. Java 字节码扫描需要 JDK，但正则快速扫描不需要

---

### 问题 5：报告验证失败

**症状**：
- `validation.issues: ["详细条目数 (0) < 总发现数 (36)"]`
- `findings_cleaned: false`

**原因**：
- 详细发现没有被插入到报告中
- 模板占位符 `<!-- DETAILED_FINDINGS_PLACEHOLDER -->` 未被替换

**解决方案**：
1. 删除旧报告文件，重新生成：
   ```bash
   rm -f audit_report*.md 审计报告*.md
   python skill.py finalize_report --output=新报告.md
   ```
2. 检查 md 文件格式是否正确（使用英文冒号）
3. 确认 `load_all_findings()` 能正确解析 md 文件
