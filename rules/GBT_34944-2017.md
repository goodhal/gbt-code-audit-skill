# GB/T 34944-2017 Java语言源代码漏洞测试规范

基于 GB/T 34944-2017《信息安全技术 Java语言源代码漏洞测试规范》的安全审计规则集，专门针对 Java 语言，与 CWE 体系对应

**标准**: GB/T 34944-2017
**语言**: java
**规则数量**: 44

## 10章 错误处理

### 问题分类GB/T34944-10.1 捕获异常后静默忽略 🟡

**严重级别**: MEDIUM | **CWE**: [CWE-390](https://cwe.mitre.org/data/definitions/390.html)


catch 块为空或仅打印而不处理异常，掩盖程序错误


**审计要点**:

检查异常处理：是否有catch(Exception e){}空catch块；是否有catch后只打印e.printStackTrace()不向上传递；安全相关异常是否被静默忽略


**修复建议**:

不允许空catch块；安全异常（认证失败/权限拒绝）必须记录日志并返回适当的错误响应；使用@ControllerAdvice统一处理

### 问题分类GB/T34944-10.2 错误信息暴露系统细节 🟡

**严重级别**: MEDIUM | **CWE**: [CWE-209](https://cwe.mitre.org/data/definitions/209.html)


异常堆栈、数据库错误信息、内部路径等细节直接返回给客户端


**审计要点**:

检查错误响应：生产环境是否有异常堆栈返回给客户端；错误响应是否包含SQL错误信息/表名/字段名；是否暴露了服务器版本/框架信息


**修复建议**:

@ControllerAdvice全局异常处理统一返回通用错误码；隐藏Server/X-Powered-By响应头；内部异常详情只记录服务端日志

## 11章 代码质量

### 问题分类GB/T34944-11.1 空指针解引用 🟡

**严重级别**: MEDIUM | **CWE**: [CWE-476](https://cwe.mitre.org/data/definitions/476.html)


方法返回值未判空即直接使用，或链式调用无空检查，可能抛出 NullPointerException


**审计要点**:

检查空指针：方法返回值是否在使用前判空；链式调用obj.getA().getB().getC()是否有NPE风险；集合get()返回null时是否有处理


**修复建议**:

使用Optional包装可能为null的返回值；添加Objects.requireNonNull()前置校验；使用@NonNull/@Nullable注解文档化

### 问题分类GB/T34944-11.2 资源未关闭泄漏 🟡

**严重级别**: MEDIUM | **CWE**: [CWE-404](https://cwe.mitre.org/data/definitions/404.html)


InputStream、Connection、Statement 等可关闭资源未在 finally 块或 try-with-resources 中关闭


**审计要点**:

检查资源泄漏：数据库Connection/Statement/ResultSet是否确保关闭；IO流是否使用try-with-resources；网络连接是否有超时和关闭处理


**修复建议**:

使用try-with-resources：try(Connection conn = ds.getConnection(); PreparedStatement ps = conn.prepareStatement(sql)) {...}

## 12章 封装

### 问题分类GB/T34944-12.1 不安全的 Java 反序列化 🔴

**严重级别**: CRITICAL | **CWE**: [CWE-502](https://cwe.mitre.org/data/definitions/502.html)


ObjectInputStream.readObject() 处理不可信数据，可被构造恶意序列化对象实现 RCE


**审计要点**:

检查不安全反序列化：是否使用ObjectInputStream反序列化来自网络/用户的数据；是否有RMI/JMX等暴露的序列化端点；是否引入了ysoserial可利用的Gadget库


**修复建议**:

使用ValidatingObjectInputStream限制可反序列化的类白名单；用JSON/XML替代Java序列化；升级受影响库（Commons-Collections/Spring/etc）；禁用不必要的RMI/JMX端口

### 问题分类GB/T34944-12.2 敏感字段序列化泄露 🟡

**严重级别**: MEDIUM | **CWE**: [CWE-213](https://cwe.mitre.org/data/definitions/213.html)


含有密码、密钥等敏感字段的 Java Bean 实现了 Serializable 但未将敏感字段标记为 transient


**审计要点**:

检查敏感字段序列化：实现Serializable的类是否有password/secret/key等敏感字段未加transient；序列化后是否可能泄露敏感信息


**修复建议**:

敏感字段加transient修饰：private transient String password；实现自定义writeObject()方法跳过或加密敏感字段

## 13章 环境

### 问题分类GB/T34944-13.1 JNDI 注入 🔴

**严重级别**: CRITICAL | **CWE**: [CWE-74](https://cwe.mitre.org/data/definitions/74.html)


InitialContext.lookup() 或日志框架的 JNDI 查找地址由用户控制，可触发远程代码执行（如 Log4Shell）


**审计要点**:

检查JNDI注入：InitialContext.lookup()的参数是否包含用户输入；Log4j版本是否<2.17.1（Log4Shell漏洞）；是否存在用户输入进入日志打印的场景


**修复建议**:

禁止JNDI查找地址来自用户输入；Log4j升级至2.17.1+（2.x）或2.12.4+（2.12.x）；设置log4j2.formatMsgNoLookups=true；禁用不必要的JNDI功能

### 问题分类GB/T34944-13.2 反射滥用（类名用户控制） 🟠

**严重级别**: HIGH | **CWE**: [CWE-470](https://cwe.mitre.org/data/definitions/470.html)


Class.forName(userInput).newInstance() 中类名由用户输入控制，可实例化任意类


**审计要点**:

检查反射滥用：Class.forName()的参数是否来自用户输入；动态加载类的类名是否有白名单限制；ClassLoader.loadClass()是否可加载任意路径的类


**修复建议**:

使用白名单Map限制可实例化的类：Map<String,Class<?>> whitelist; 禁止任意类名传入反射API

## 14章 Web安全

### 问题分类GB/T34944-14.1 JSP/模板 XSS 输出未编码 🟠

**严重级别**: HIGH | **CWE**: [CWE-79](https://cwe.mitre.org/data/definitions/79.html)


JSP 使用 <%=request.getParameter()%> 或 EL 表达式 ${param.x} 未经 HTML 编码直接输出


**审计要点**:

检查JSP XSS：JSP中是否使用<%=request.getParameter()%>直接输出；${param.x}在普通JSP中是否未经JSTL的fn:escapeXml()处理；Thymeleaf是否使用th:utext渲染用户数据


**修复建议**:

使用<c:out value='${param.x}'/>或OWASP Java Encoder的Encode.forHtml()；Thymeleaf使用th:text（自动编码）替代th:utext

### 问题分类GB/T34944-14.2 开放重定向 🟡

**严重级别**: MEDIUM | **CWE**: [CWE-601](https://cwe.mitre.org/data/definitions/601.html)


response.sendRedirect() 的目标 URL 由用户控制且未做白名单校验，可用于钓鱼


**审计要点**:

检查开放重定向：response.sendRedirect()的URL参数是否来自请求参数；是否验证了目标URL在允许的域名白名单内；OAuth回调URL是否严格验证


**修复建议**:

白名单验证目标域名；使用相对路径重定向；OAuth redirect_uri严格注册验证；若需外部跳转，显示中间确认页面

### 问题分类GB/T34944-14.3 SSRF（服务端请求伪造） 🟠

**严重级别**: HIGH | **CWE**: [CWE-918](https://cwe.mitre.org/data/definitions/918.html)


new URL(userInput).openConnection() 或 HttpClient 发起请求时 URL 由用户提供，可访问内网资源


**审计要点**:

检查SSRF：HTTP请求的目标URL是否来自用户输入；是否可访问内网IP（10.x/172.16.x/192.168.x）；是否可访问云元数据（169.254.169.254）；file://等危险协议是否被禁止


**修复建议**:

URL白名单验证（精确域名）；DNS解析后检查IP是否在内网段；禁止file://、gopher://等协议；通过受控代理转发外部请求

### 问题分类GB/T34944-14.4 不安全的文件上传 🔴

**严重级别**: CRITICAL | **CWE**: [CWE-434](https://cwe.mitre.org/data/definitions/434.html)


文件上传仅检查扩展名，未验证 MIME 类型魔数，上传目录在 Web 根路径且有执行权限


**审计要点**:

检查文件上传安全：是否仅检查文件扩展名而非MIME类型；上传目录是否在webroot下；是否限制了文件大小；上传文件名是否使用用户提供的原名；是否防止了双扩展名绕过


**修复建议**:

读取文件头魔数验证类型；上传目录禁止执行权限且在webroot外；文件UUID重命名；大小限制（如10MB）；仅允许白名单文件类型

### 问题分类GB/T34944-14.5 危险 HTTP 方法未禁用 🟢

**严重级别**: LOW | **CWE**: [CWE-749](https://cwe.mitre.org/data/definitions/749.html)


Web 服务器未禁用 TRACE/TRACK/PUT/DELETE 等危险 HTTP 方法


**审计要点**:

检查HTTP方法限制：服务器是否响应OPTIONS请求并暴露TRACE/PUT/DELETE方法；TRACE方法是否可用（XST攻击风险）


**修复建议**:

Web服务器（Nginx/Tomcat）配置只允许GET/POST/HEAD等必要方法；禁用TRACE方法；REST API按需开放PUT/DELETE并加权限控制

### 问题分类GB/T34944-14.6 点击劫持防护缺失 🟡

**严重级别**: MEDIUM | **CWE**: [CWE-1021](https://cwe.mitre.org/data/definitions/1021.html)


未设置 X-Frame-Options 或 Content-Security-Policy frame-ancestors，页面可被嵌入 iframe 被点击劫持


**审计要点**:

检查点击劫持防护：响应头是否设置了X-Frame-Options: DENY/SAMEORIGIN；CSP是否设置了frame-ancestors；页面是否可被恶意嵌入iframe


**修复建议**:

设置X-Frame-Options: DENY或SAMEORIGIN；或使用Content-Security-Policy: frame-ancestors 'self'；禁止页面被iframe嵌入

## 6.1 SQL注入

### 问题分类GB/T34944-6.1.1 Statement 字符串拼接 SQL 注入 🔴

**严重级别**: CRITICAL | **CWE**: [CWE-89](https://cwe.mitre.org/data/definitions/89.html)


使用 Statement.execute()/executeQuery() 直接拼接用户输入构造 SQL 语句


**审计要点**:

检查SQL注入漏洞：是否使用Statement拼接SQL；'SELECT * FROM t WHERE id=' + userId 此类模式；动态构建的SQL字符串中是否含有用户可控参数


**修复建议**:

使用PreparedStatement参数化查询：Connection.prepareStatement('SELECT * FROM t WHERE id=?')，通过setXxx()传参；禁止字符串拼接SQL

### 问题分类GB/T34944-6.1.2 MyBatis ${}参数注入 🔴

**严重级别**: CRITICAL | **CWE**: [CWE-89](https://cwe.mitre.org/data/definitions/89.html)


MyBatis XML Mapper 或注解中使用 ${} 传入用户可控参数导致 SQL 注入


**审计要点**:

检查MyBatis注入：Mapper XML文件或@Select注解中是否使用${}传递用户输入参数；排序字段/动态表名是否未做白名单处理


**修复建议**:

全面使用#{}占位符替代${}；动态表名/列名场景使用Java代码白名单Map映射后再传入

### 问题分类GB/T34944-6.1.3 JPA/Hibernate HQL 注入 🟠

**严重级别**: HIGH | **CWE**: [CWE-89](https://cwe.mitre.org/data/definitions/89.html)


HQL/JPQL 查询字符串中直接拼接用户输入，未使用命名参数或位置参数


**审计要点**:

检查HQL注入：entityManager.createQuery()或session.createQuery()中是否存在字符串拼接；是否使用了命名参数:paramName或?1位置参数


**修复建议**:

使用命名参数：createQuery('FROM User WHERE name = :name').setParameter('name', input)；或使用Criteria API

## 6.2 命令注入

### 问题分类GB/T34944-6.2.1 Runtime.exec 命令注入 🔴

**严重级别**: CRITICAL | **CWE**: [CWE-78](https://cwe.mitre.org/data/definitions/78.html)


Runtime.getRuntime().exec() 中使用字符串形式传入包含用户输入的命令


**审计要点**:

检查命令注入：Runtime.getRuntime().exec(cmd)中cmd是否包含用户输入；是否使用单字符串形式exec(String)而非安全的exec(String[])；是否有分号/管道符注入风险


**修复建议**:

使用数组形式传参：exec(new String[]{'/bin/sh', '-c', sanitizedCmd})；严格白名单过滤允许的命令参数；优先使用Java库替代系统命令

### 问题分类GB/T34944-6.2.2 ProcessBuilder 命令注入 🔴

**严重级别**: CRITICAL | **CWE**: [CWE-78](https://cwe.mitre.org/data/definitions/78.html)


ProcessBuilder 的命令参数列表中包含未过滤的用户输入


**审计要点**:

检查ProcessBuilder注入：ProcessBuilder构造参数中是否有用户输入；command(List)中的参数是否经过白名单过滤；是否有shell元字符注入风险


**修复建议**:

对每个参数进行白名单验证；避免将用户输入作为命令本身（只允许作为明确定义的参数）；使用正则表达式验证参数格式

## 6.3 路径遍历

### 问题分类GB/T34944-6.3.1 文件路径遍历 🟠

**严重级别**: HIGH | **CWE**: [CWE-22](https://cwe.mitre.org/data/definitions/22.html)


文件路径由用户输入构造，未过滤 ../ 序列，可访问系统任意文件


**审计要点**:

检查路径遍历：new File(baseDir + userInput)是否验证了路径合法性；是否过滤了../和URL编码的%2e%2e；getCanonicalPath()后是否检查是否以基础目录开头


**修复建议**:

使用canonicalPath规范化后检查前缀：if(!canonical.startsWith(base.getCanonicalPath())) throw SecurityException；禁止../序列

### 问题分类GB/T34944-6.3.2 ZIP 解压路径穿越（Zip Slip） 🟠

**严重级别**: HIGH | **CWE**: [CWE-22](https://cwe.mitre.org/data/definitions/22.html)


解压 ZIP/TAR 文件时未验证条目路径，恶意压缩包可覆写任意文件


**审计要点**:

检查Zip Slip：解压ZIP/TAR时是否验证ZipEntry.getName()不包含../；解压目标路径是否验证在允许目录内


**修复建议**:

解压前验证每个entry路径：File destFile = new File(destDir, entry.getName()); if(!destFile.getCanonicalPath().startsWith(destDir.getCanonicalPath())) throw exception;

## 6.4 XML外部实体注入

### 问题分类GB/T34944-6.4.1 DocumentBuilder XXE 注入 🟠

**严重级别**: HIGH | **CWE**: [CWE-611](https://cwe.mitre.org/data/definitions/611.html)


DocumentBuilderFactory 未禁用外部实体解析，XML 可读取本地文件或发起 SSRF


**审计要点**:

检查XXE漏洞：DocumentBuilderFactory是否设置了disallow-doctype-decl特性；是否关闭了外部一般实体和外部参数实体；是否设置了FEATURE_SECURE_PROCESSING


**修复建议**:

dbf.setFeature('http://apache.org/xml/features/disallow-doctype-decl', true); dbf.setFeature(XMLConstants.FEATURE_SECURE_PROCESSING, true); dbf.setXIncludeAware(false); dbf.setExpandEntityReferences(false);

### 问题分类GB/T34944-6.4.2 SAXParser/XMLReader XXE 注入 🟠

**严重级别**: HIGH | **CWE**: [CWE-611](https://cwe.mitre.org/data/definitions/611.html)


SAXParser 或 XMLReader 未禁用外部实体，存在 XXE 漏洞


**审计要点**:

检查SAXParser XXE：SAXParserFactory或XMLReader是否设置了FEATURE_SECURE_PROCESSING；是否禁用了external-general-entities和external-parameter-entities特性


**修复建议**:

saxFactory.setFeature(XMLConstants.FEATURE_SECURE_PROCESSING, true); xmlReader.setFeature('http://xml.org/sax/features/external-general-entities', false);

## 7.1 不安全随机数

### 问题分类GB/T34944-7.1.1 弱随机数用于安全场景 🟠

**严重级别**: HIGH | **CWE**: [CWE-338](https://cwe.mitre.org/data/definitions/338.html)


java.util.Random 或 Math.random() 被用于生成安全相关的随机数（Token、密钥、验证码）


**审计要点**:

## 7.2 弱加密算法

### 问题分类GB/T34944-7.2.1 使用弱对称加密算法 🟠

**严重级别**: HIGH | **CWE**: [CWE-327](https://cwe.mitre.org/data/definitions/327.html)


使用 DES、3DES、RC4 等已被破解或不推荐的对称加密算法


**审计要点**:

检查弱对称加密：Cipher.getInstance()参数是否为DES/RC2/RC4/3DES/Blowfish；加密算法是否是国密推荐算法或AES


**修复建议**:

使用AES/GCM/NoPadding（推荐）或AES/CBC/PKCS5Padding；禁用DES/3DES/RC4；若使用CBC模式，IV必须随机生成

### 问题分类GB/T34944-7.2.2 使用弱哈希算法存储密码 🟠

**严重级别**: HIGH | **CWE**: [CWE-328](https://cwe.mitre.org/data/definitions/328.html)


密码使用 MD5 或 SHA-1 哈希存储，可被彩虹表或暴力破解


**审计要点**:

检查密码存储：密码是否使用MD5/SHA-1/SHA-256等哈希（无加盐）存储；是否使用了专为密码设计的哈希函数


**修复建议**:

密码存储使用BCryptPasswordEncoder（工作因子≥12）、PBKDF2WithHmacSHA256或Argon2；哈希算法用SHA-256/SHA-3时必须加随机盐

### 问题分类GB/T34944-7.2.3 AES ECB 模式使用 🟠

**严重级别**: HIGH | **CWE**: [CWE-327](https://cwe.mitre.org/data/definitions/327.html)


AES 使用 ECB 模式加密，ECB 模式不提供语义安全（相同明文产生相同密文）


**审计要点**:

检查AES ECB模式：Cipher.getInstance()是否使用AES/ECB/PKCS5Padding；ECB模式无法隐藏明文数据模式


**修复建议**:

使用AES/GCM/NoPadding（推荐，提供认证加密）或AES/CBC/PKCS5Padding（IV必须随机生成）

### 问题分类GB/T34944-7.2.4 硬编码加密密钥或 IV 🔴

**严重级别**: CRITICAL | **CWE**: [CWE-321](https://cwe.mitre.org/data/definitions/321.html)


加密密钥或初始化向量（IV）为代码中的硬编码字符串常量


**审计要点**:

检查硬编码密钥：源代码中是否有byte[] key = {...}或String key = '...'等硬编码加密密钥；IV是否使用固定值或全零


**修复建议**:

密钥从KMS或安全配置中心加载；IV每次加密随机生成（SecureRandom）并与密文一同存储传输

## 7.3 不安全SSL/TLS

### 问题分类GB/T34944-7.3.1 空 TrustManager（信任所有证书） 🔴

**严重级别**: CRITICAL | **CWE**: [CWE-295](https://cwe.mitre.org/data/definitions/295.html)


X509TrustManager 的 checkServerTrusted() 方法为空实现，信任所有 SSL 证书包括无效/自签名证书


**审计要点**:

检查SSL证书验证：是否实现了空的X509TrustManager（checkServerTrusted为空方法）；是否使用了TrustAllCerts模式；是否有TRUST_ALL相关代码


**修复建议**:

使用系统默认TrustManager；不要实现空的TrustManager；生产环境不允许自签名证书（或配置证书固定Certificate Pinning）

### 问题分类GB/T34944-7.3.2 禁用 Hostname 验证 🔴

**严重级别**: CRITICAL | **CWE**: [CWE-297](https://cwe.mitre.org/data/definitions/297.html)


HttpsURLConnection.setHostnameVerifier() 设置为永远返回 true，不验证服务器域名


**审计要点**:

检查Hostname验证：是否设置了自定义HostnameVerifier永远返回true；OkHttp/Apache HttpClient等是否关闭了hostname验证


**修复建议**:

使用默认HostnameVerifier；删除自定义的AllowAllHostnameVerifier；生产环境不得关闭主机名验证

### 问题分类GB/T34944-7.3.3 使用弱 TLS 版本 🟠

**严重级别**: HIGH | **CWE**: [CWE-326](https://cwe.mitre.org/data/definitions/326.html)


使用 SSLv3、TLSv1.0 或 TLSv1.1 等已弃用的协议版本


**审计要点**:

检查TLS版本：SSLContext.getInstance()参数是否为SSL/TLSv1/TLSv1.1；服务器配置是否支持TLS 1.0或1.1；是否使用了POODLE/BEAST等已知漏洞协议


**修复建议**:

使用SSLContext.getInstance('TLSv1.3')或最低TLSv1.2；配置服务器禁用SSLv3/TLS1.0/TLS1.1；使用强加密套件

## 8.1 身份鉴别与会话管理

### 问题分类GB/T34944-8.1.1 Session 固定攻击 🟠

**严重级别**: HIGH | **CWE**: [CWE-384](https://cwe.mitre.org/data/definitions/384.html)


登录成功后未重新生成 Session ID，攻击者可预先设置 Session ID 后等待受害者登录


**审计要点**:

检查Session固定：登录成功后是否调用request.getSession(false).invalidate()并重新创建Session；是否存在Session ID可被URL参数控制的情况


**修复建议**:

登录成功后立即重建Session：session.invalidate(); HttpSession newSession = request.getSession(true); // 迁移需要的属性

### 问题分类GB/T34944-8.1.2 会话未设置超时 🟡

**严重级别**: MEDIUM | **CWE**: [CWE-613](https://cwe.mitre.org/data/definitions/613.html)


Session 超时时间未设置或设置为永不超时，增加 Session 劫持风险


**审计要点**:

检查会话超时：web.xml中session-timeout是否设置为-1（永不超时）或过大值；是否有活跃超时和绝对超时双重控制


**修复建议**:

设置合理超时：<session-timeout>30</session-timeout>（30分钟）；同时实施绝对超时（如8小时）；敏感操作前重新验证身份

### 问题分类GB/T34944-8.1.3 Cookie 缺少安全属性 🟡

**严重级别**: MEDIUM | **CWE**: [CWE-614](https://cwe.mitre.org/data/definitions/614.html)


Session Cookie 未设置 HttpOnly 或 Secure 属性，易被 XSS 窃取或通过 HTTP 传输


**审计要点**:

检查Cookie安全属性：Session Cookie是否设置了HttpOnly防止XSS窃取；HTTPS环境下是否设置了Secure属性；是否设置了SameSite属性


**修复建议**:

Cookie设置：cookie.setHttpOnly(true); cookie.setSecure(true); 在web.xml配置<cookie-config><http-only>true</http-only><secure>true</secure></cookie-config>

### 问题分类GB/T34944-8.1.4 明文存储密码 🔴

**严重级别**: CRITICAL | **CWE**: [CWE-256](https://cwe.mitre.org/data/definitions/256.html)


配置文件或数据库中使用明文或可逆加密存储用户密码


**审计要点**:

检查密码存储：数据库password字段是否存储明文或Base64编码；是否使用了可逆加密（AES加密密码）而非单向哈希；properties配置文件中是否有明文密码


**修复建议**:

使用BCrypt：new BCryptPasswordEncoder().encode(rawPassword)；Spring Security配置BCryptPasswordEncoder；迁移存量明文密码

## 8.2 访问控制

### 问题分类GB/T34944-8.2.1 缺少方法级权限控制 🟠

**严重级别**: HIGH | **CWE**: [CWE-285](https://cwe.mitre.org/data/definitions/285.html)


Controller 方法没有权限注解，或权限控制仅依赖 URL 模式匹配


**审计要点**:

检查方法级权限：Controller方法是否有@PreAuthorize/@Secured/@RolesAllowed注解；是否开启了全局方法安全（@EnableMethodSecurity）；是否有接口可绕过URL拦截直接访问


**修复建议**:

开启@EnableMethodSecurity(prePostEnabled=true)；关键方法加@PreAuthorize('hasRole(ADMIN)')注解；不单独依赖URL模式配置权限

### 问题分类GB/T34944-8.2.2 不安全的直接对象引用（IDOR） 🔴

**严重级别**: CRITICAL | **CWE**: [CWE-639](https://cwe.mitre.org/data/definitions/639.html)


接口通过 userId/orderId 等参数直接访问资源，未在服务端验证资源归属关系


**审计要点**:

检查IDOR：获取/修改/删除资源的接口是否验证资源属于当前登录用户；是否可以通过修改参数中的ID访问他人数据；Spring Data查询是否加了userId条件


**修复建议**:

查询时加入当前用户约束：repository.findByIdAndUserId(id, currentUserId)；使用@PostAuthorize注解验证返回对象所属；关键操作记录审计日志

### 问题分类GB/T34944-8.2.3 Spring Security 配置过宽 🟠

**严重级别**: HIGH | **CWE**: [CWE-284](https://cwe.mitre.org/data/definitions/284.html)


Spring Security 配置放行所有请求，或全局禁用 CSRF 防护


**审计要点**:

检查Spring Security配置：是否有antMatchers('/**').permitAll()放行所有请求；csrf().disable()是否在非API场景使用；是否有接口未加认证要求


**修复建议**:

最小化permitAll范围；API服务使用JWT+无状态Session时可关闭CSRF但需确保Token不在Cookie中；定期审查Security配置

## 8.3 CSRF防护

### 问题分类GB/T34944-8.3.1 CSRF 防护缺失 🟠

**严重级别**: HIGH | **CWE**: [CWE-352](https://cwe.mitre.org/data/definitions/352.html)


POST/PUT/DELETE 等状态变更接口未验证 CSRF Token，可被跨站请求伪造利用


**审计要点**:

检查CSRF防护：Spring Security是否全局禁用了CSRF；表单提交是否包含CSRF Token；Cookie是否缺少SameSite属性；AJAX请求是否携带CSRF Token


**修复建议**:

保持Spring Security CSRF默认开启；前后端分离时使用Double Submit Cookie或X-CSRF-Token请求头；设置Cookie SameSite=Lax

## 9章 时间与状态

### 问题分类GB/T34944-9.1 竞态条件（TOCTOU） 🟠

**严重级别**: HIGH | **CWE**: [CWE-367](https://cwe.mitre.org/data/definitions/367.html)


检查（Time-of-Check）与使用（Time-of-Use）之间缺乏原子性保证，可被并发利用


**审计要点**:

检查竞态条件：if(file.exists())后再操作文件的模式；余额检查后扣款之间是否有锁；高并发场景下的先查后改操作是否原子


**修复建议**:

使用数据库乐观锁/悲观锁；文件操作使用Files.move()等原子API；账户扣款使用update带WHERE条件CAS操作

### 问题分类GB/T34944-9.2 线程不安全的共享对象 🟠

**严重级别**: HIGH | **CWE**: [CWE-362](https://cwe.mitre.org/data/definitions/362.html)


SimpleDateFormat、HashMap 等非线程安全类作为 static 共享变量在多线程中使用


**审计要点**:

检查线程安全：是否有static SimpleDateFormat共享实例；HashMap作为类级别缓存是否有并发写；单例Bean中是否有可变实例变量


**修复建议**:

使用ThreadLocal<SimpleDateFormat>或DateTimeFormatter（Java 8+线程安全）；使用ConcurrentHashMap；无状态Bean设计

## 误报排除指南


以下情况通常为误报，不需要修复：


- **工具类/工具函数**：DateUtil、StringUtil、JSONUtil 等通用工具类的方法本身不是漏洞
- **测试代码**：单元测试、集成测试中的代码通常不需要修复
- **示例代码**：Demo、Example 等示例代码
- **配置类**：Properties/yml 配置加载通常不涉及用户输入
- **常量定义**：final static 常量是安全的
- **白名单验证**：已有白名单校验的代码片段
