# 审计工作流程

本文档详细描述行号验证和防幻觉机制。

> **LLM 审计独立性原则**：见 [SKILL.md](../SKILL.md) "LLM 审计强制要求"章节

---

## 行号验证流程

```
┌─────────────────────────────────────────────────────────────────────┐
│                    LLM 审计行号验证流程                              │
├─────────────────────────────────────────────────────────────────────┤
│  1️⃣ 读取源文件  ◀ 【必须】使用 Read 工具读取完整源文件              │
│  2️⃣ 定位问题代码  ◀ 【必须】在源文件中精确搜索问题代码片段          │
│  3️⃣ 确认行号  ◀ 【必须】验证行号对应的是代码行而非注释行            │
│  4️⃣ 记录发现  ◀ 【必须】行号验证通过后才创建 md 文件               │
└─────────────────────────────────────────────────────────────────────┘
```

**强制要求**：
- 必须逐行核对源文件确认漏洞位置
- 行号必须精确对应问题代码所在行
- 禁止根据注释推断行号，必须定位实际代码
- 必须使用 Grep 工具精确查找行号

## 防幻觉工作流程

```
┌─────────────────────────────────────────────────────────────────────┐
│                    LLM 审计防幻觉流程                               │
├─────────────────────────────────────────────────────────────────────┤
│  1️⃣ 创建md文件  ◀ LLM写入审计发现到 findings/llm_audit/XXX.md     │
│  2️⃣ 调用验证    ◀ python skill.py validate_finding XXX.md         │
│  3️⃣ 验证结果    ◀ 成功 → 编号+1，开始下一个发现                    │
│               ◀ 失败 → 尝试修正行号 → 成功则通过，失败则跳过        │
└─────────────────────────────────────────────────────────────────────┘
```

**必须逐个验证**：
- 禁止批量验证
- 创建一个md → 验证 → 通过后再创建下一个
- 验证失败时：尝试修正行号，超过2次则跳过该问题

---

## 常见幻觉问题

| 幻觉类型 | 错误示例 | 正确做法 |
|---------|---------|---------|
| 注释行号 | 报告行号27，实际是注释行 | 必须定位到实际代码行 |
| 行号偏移 | 报告行号150，实际代码在133 | 使用Grep精确搜索 |
| 代码不存在 | 报告的代码片段在源文件中不存在 | 必须在源文件中验证存在 |
| 重复行号 | 多个漏洞使用相同行号 | 每个漏洞必须有唯一行号 |
